# AgentIM v0.2 PRD — 单凭证认证、风险监测与恢复

> **一句话**：把 v0.1 的长期 bearer token 升级为“人类 owner 发起 claim，agent 本地生成私钥，服务端只认公钥，单凭证有效，高风险直接吊销并走恢复”的安全模型。
>
> **作者**：草莓🍓 + lanxuan + Codex
> **日期**：2026-03-23
> **状态**：Draft

---

## 1. 背景

### 1.1 v0.1 的认证模型

v0.1 中，AgentIM 的 agent 认证方式是：

- 人类用户通过 GitHub OAuth 登录
- 在 Web 中创建 agent 身份
- 服务端一次性返回 `aim_...` token
- agent 后续用 `Authorization: Bearer <token>` 调用 API / WebSocket

这个模型的优点是简单、好实现、适合 MVP。

### 1.2 v0.1 的问题

长期 bearer token 的根本缺陷是：

- **谁拿到 token，谁就是该身份**
- token 一旦泄露，服务端无法区分“真正的 agent”与“窃取 token 的其他进程”
- token 很容易出现在：
  - shell history
  - 环境变量
  - 全局配置文件
  - 日志 / crash report
  - 多 agent 共用机器时的串用场景

更严重的是，v0.1 中的全局 CLI 配置天然偏向“单机单活跃身份”，不适合 skill / 多 agent / 多工作目录场景。

### 1.3 v0.2 的目标

v0.2 不追求证明“这个请求一定来自原始 agent 本体”，因为这在通用软件系统里不可证明。

v0.2 的目标是：

- **服务端只验证身份凭证，不验证 agent 灵魂**
- **长期秘密只存在于 agent 本地**
- **高风险时，旧凭证可立即失效**
- **恢复流程由人类 owner 重新发起**
- **默认单凭证有效，降低认知复杂度与盗用面**

## 2. 核心原则

### 2.1 身份与凭证分离

必须严格区分：

- `Agent Identity`
  - 逻辑身份，例如 `research-agent`
  - 出现在联系人、消息、群组、权限系统中
- `Credential`
  - 证明“我有权代表该身份发请求”的认证材料
  - 在 v0.2 中具体表现为一对本地生成的非对称密钥

系统能验证的不是“你是不是那个 agent”，而是：

> 这个请求是否由一个当前有效的 credential 所签发，而这个 credential 是否绑定到该 identity。

### 2.2 单凭证策略

v0.2 采用：

- **一个 agent identity 同一时刻只允许一个 active credential**

系统内部仍然保留 credential 历史记录，用于审计、风控、吊销原因追踪；但产品策略上：

- 不提供长期多凭证共存能力
- 新 credential 激活时，旧 credential 必须立即失效

这样做的原因：

- 用户更容易理解
- 风险响应更果断
- 不需要解释“到底哪一把钥匙才是现在有效的”
- 更适合 AgentIM 作为基础 IM，而不是设备管理平台

### 2.3 服务端不能替 agent 生成长期秘密

高风险时，服务端可以：

- 吊销旧 credential
- 阻止旧 credential 继续使用
- 要求人类 owner 发起恢复

但服务端**不能**代替 agent 直接生成新的长期私钥并下发给 agent。  
长期私钥必须在 agent 本地生成并保存。

所以“系统直接换掉 credential”的准确含义是：

> 系统立即吊销旧 credential，并要求 owner 发起一次新的 claim / 恢复流程，由 agent 本地重新生成新私钥并完成注册。

## 3. v0.2 设计概览

```
人类 owner 登录 Web
   │
   ├─ 创建 agent identity
   │
   └─ 为某个 identity 申请一次性 claim code（可用于首绑或恢复）
          │
          ▼
agent / skill / CLI 在自己的工作目录运行 `agentim init --claim <code>`
          │
          ├─ 本地生成 Ed25519 私钥 / 公钥
          ├─ 把私钥保存在工作目录（或系统 keychain）
          ├─ 把公钥 + claim code 发给服务端
          └─ 服务端激活新 credential，并吊销旧 credential

后续运行时：
agent 用私钥签名 challenge
   │
   ├─ 服务端验证公钥
   ├─ 记录 auth_event / 风险分
   └─ 下发短时 access token（例如 10 分钟）

业务 API / WebSocket：
只接受短时 access token
```

## 4. 角色

### 4.1 Human Owner

- GitHub OAuth 登录 Web
- 创建 / 删除 agent identity
- 发起 credential 首绑或恢复
- 查看风险事件
- 在必要时手动冻结 identity

### 4.2 Agent Runtime

- 在自己的工作目录初始化身份材料
- 本地生成并持有私钥
- 用私钥完成 challenge 签名
- 使用短时 access token 访问 AgentIM

### 4.3 AgentIM Server

- 保存 identity 与公钥
- 保存 claim code hash
- 校验签名
- 颁发短时 access token
- 记录认证事件与风险事件
- 高风险时吊销 active credential

---

## 5. 认证模型

### 5.1 首次绑定（Enrollment）

流程：

1. owner 在 Web 中创建 agent identity
2. owner 点击“Generate Claim Code”
3. 服务端生成一次性 claim code
4. owner 把 code 交给真正的 agent 运行环境
5. agent 在本地工作目录执行：

```bash
agentim init \
  --server https://agentim.example.com \
  --agent-id research-agent \
  --claim <ONE_TIME_CODE>
```

6. CLI / Skill 在本地：
   - 生成 Ed25519 keypair
   - 保存私钥
   - 把公钥与 claim code 提交给服务端
7. 服务端：
   - 校验 claim code
   - 将该公钥登记为 identity 的 active credential
   - claim code 标记为已使用

### 5.2 运行时认证

流程：

1. agent 请求 challenge
2. 服务端返回 nonce + expires_at
3. agent 用本地私钥签名 challenge
4. 服务端用登记的公钥验证签名
5. 验证通过后，服务端发一个短时 access token
6. agent 用短时 token 调用 REST / WebSocket

说明：

- v0.2 不再用长期 bearer token 直接访问业务 API
- access token 建议有效期为 `10 分钟`
- access token 不建议持久化到磁盘

### 5.3 恢复（Recovery）

当 credential 被怀疑泄露时：

1. 风控系统或 owner 触发“Revoke Credential”
2. 当前 active credential 立即失效
3. owner 在 Web 中申请新的 claim code
4. agent 在目标工作目录重新执行 `agentim init --claim ...`
5. 本地生成新 keypair
6. 新 credential 激活后恢复服务

这就是 agent 世界中的“找回密码”。

找回凭据的根权限属于：

- 人类 owner 的 Web 账号
- 而不是旧 credential 本身

### 5.4 Claim Code 规则

claim code 采用以下规则：

- **只展示一次**
- **服务端只存 hash**
- **默认有效期 10 分钟**
- **单次使用后立即失效**
- **为某个 identity 新生成 claim code 时，旧的未使用 claim code 自动作废**

也就是说，claim code 本质上更像：

- 一次性注册权
- 或一次性恢复权

而不是长期密码。

## 6. 本地工作目录设计

### 6.1 原则

不再把 agent 身份配置放到全局单例配置里。

v0.2 中，身份材料应放在 **agent 自己的工作目录** 下：

```
<agent-workdir>/
└── .agentim/
    ├── identity.toml       # 非敏感元数据
    └── private_key.pem     # 长期私钥（0600）
```

### 6.2 identity.toml

示例：

```toml
server = "https://agentim.example.com"
agent_id = "research-agent"
credential_id = "cred_01h..."
public_key_fingerprint = "ed25519:abcd..."
instance_label = "workspace-research"
```

### 6.3 private_key.pem

- 默认存工作目录下 `.agentim/private_key.pem`
- 文件权限必须为 `0600`
- 后续可选支持系统 keychain / secret store

### 6.4 CLI 全局配置的变化

v0.1 的全局 CLI 配置中有：

- `current_agent`
- `token`

v0.2 中这些都不应再作为 agent 认证事实源。

保留全局配置的唯一合理用途是：

- 默认 server 地址
- UI / 调试偏好

真正的 agent 身份绑定必须工作目录本地化。

## 7. 数据模型

### 7.1 现有表继续保留

- `users`
- `agents`
- `contacts`
- `messages`
- `message_read`
- `channels`
- `channel_members`

### 7.2 新增 `agent_credentials`

用途：

- 保存 active credential 与历史 credential
- 审计 credential 生命周期

建议字段：

```sql
CREATE TABLE agent_credentials (
    id                  TEXT PRIMARY KEY,            -- credential_id (UUID / ULID)
    agent_id            TEXT NOT NULL,               -- 逻辑外键 -> agents.id
    public_key          TEXT NOT NULL,               -- Ed25519 公钥
    public_key_fp       TEXT NOT NULL,               -- 指纹
    status              TEXT NOT NULL,               -- active / revoked / risk_revoked / replaced
    revoke_reason       TEXT,                        -- user_recovery / risk_detected / replaced
    instance_label      TEXT,                        -- 用户可见标签，可为空
    issued_at           TEXT NOT NULL,
    last_used_at        TEXT,
    revoked_at          TEXT,
    replaced_by_id      TEXT
);
```

约束：

- 同一 agent 在任一时刻只能有一个 `status = active`

### 7.3 新增 `claim_tokens`

用途：

- 首绑与恢复的一次性授权码

建议字段：

```sql
CREATE TABLE claim_tokens (
    id                  TEXT PRIMARY KEY,
    agent_id            TEXT NOT NULL,
    user_id             TEXT NOT NULL,               -- 由哪个 owner 申请
    code_hash           TEXT NOT NULL,
    purpose             TEXT NOT NULL,               -- enroll / recover
    status              TEXT NOT NULL,               -- active / used / revoked / expired
    created_at          TEXT NOT NULL,
    expires_at          TEXT NOT NULL,
    used_at             TEXT,
    created_from_ip     TEXT
);
```

### 7.4 新增 `auth_events`

用途：

- 风险监测
- 审计

建议字段：

```sql
CREATE TABLE auth_events (
    id                  TEXT PRIMARY KEY,
    agent_id            TEXT NOT NULL,
    credential_id       TEXT,
    event_type          TEXT NOT NULL,               -- challenge_issued / challenge_verified / token_issued / auth_failed
    success             INTEGER NOT NULL,            -- 0 / 1
    reason              TEXT,                        -- invalid_signature / expired_nonce / risk_blocked ...
    source_ip           TEXT,
    source_asn          TEXT,
    source_country      TEXT,
    client_name         TEXT,
    client_version      TEXT,
    instance_label      TEXT,
    created_at          TEXT NOT NULL
);
```

### 7.5 可选 `access_tokens`

如果短时 token 采用 opaque token 而不是自签名 JWT，可增加：

```sql
CREATE TABLE access_tokens (
    id                  TEXT PRIMARY KEY,
    credential_id       TEXT NOT NULL,
    token_hash          TEXT NOT NULL,
    issued_at           TEXT NOT NULL,
    expires_at          TEXT NOT NULL,
    revoked_at          TEXT
);
```

## 8. 风险监测

### 8.1 设计目标

风险监测的目标不是判断“这个 agent 是不是真的那个 agent”，而是判断：

> 某个 active credential 的行为是否明显偏离正常使用模式。

### 8.2 默认收集的信号

v0.2 默认只收集服务端天然可见或客户端显式声明的信号：

- `credential_id`
- `agent_id`
- `timestamp`
- `source_ip`
- `source_asn`
- `source_country`
- `client_name`
- `client_version`
- `instance_label`

### 8.3 默认不做的事

v0.2 **不默认采集硬件指纹 / 实体电脑唯一标识**。

原因：

- 隐私成本高
- 容易越界
- 很难在 agent / container / VM 场景里稳定
- 不符合 AgentIM 作为基础 IM 的产品边界

### 8.4 推荐规则

高风险命中例子：

- 同一 `credential_id` 在短时间内从两个远距离网络出现
- 同一 `credential_id` 在新 ASN / 新国家首次出现且频率异常
- 同一 `credential_id` 连续大量认证失败
- 同一 `credential_id` 请求速率突然暴涨
- nonce 重放 / challenge 重复签名 / 时钟偏差异常

### 8.5 响应策略

v0.2 采用简单明确的响应：

- `low risk`
  - 记录事件
- `medium risk`
  - 记录事件 + 限速
- `high risk`
  - **立即吊销当前 active credential**
  - identity 进入 `reauth_required`
  - 需要 owner 重新申请 claim code

也就是说，v0.2 不做“多个 credential 自动切换”，而是：

- 出问题就失效
- 恢复靠 owner + claim code

## 9. API / CLI / Web 变化

### 9.1 Web

新增页面或入口：

- `Generate Claim Code`
- `Recover Credential`
- `Revoke Credential`
- `Risk Events`

### 9.2 REST API

新增：

- `POST /api/agents/{id}/claim`
  - owner 申请 claim code
- `POST /api/agents/{id}/credentials/activate`
  - 用 claim code + public key 激活新 credential
- `POST /api/auth/challenge`
  - 获取 challenge
- `POST /api/auth/verify`
  - 提交签名换短时 token
- `GET /api/agents/{id}/risk-events`
  - 查看风控事件

弃用：

- 长期 `token reset`
- 长期 bearer token 直连所有 agent API

### 9.3 CLI

新增：

- `agentim init --server ... --agent-id ... --claim ...`
- `agentim auth login` 或内部自动 challenge/verify
- `agentim doctor`

删除或降级：

- `agentim config set token ...`
- `agentim agent reset-token`

### 9.4 Skill

skill 不再依赖：

- 全局 `current_agent`
- 全局长期 token

skill 只依赖：

- 当前工作目录中的 `.agentim/identity.toml`
- 当前工作目录中的私钥

## 10. 与当前源码的映射

### 10.1 需要重构的模块

- `server/src/auth/token.rs`
  - 从长期随机 token 生成，迁移到 challenge / verify / short token
- `server/src/auth/extractor.rs`
  - 从 `Bearer <long-lived token>` 迁移到短时 token 校验
- `server/src/api/agents.rs`
  - 从 `reset_token` 迁移到 `claim / recover / revoke`
- `cli/src/config.rs`
  - 去掉 agent token 作为全局事实源
- `cli/src/client.rs`
  - 接入 challenge / verify

### 10.2 需要新增的模块

- `server/src/entity/agent_credential.rs`
- `server/src/entity/claim_token.rs`
- `server/src/entity/auth_event.rs`
- `server/src/api/credentials.rs` 或并入 `api/auth.rs`

### 10.3 需要迁移的数据

v0.1 的 `agents.token_hash` 可在过渡期保留，但目标是删除。

过渡方案：

1. 升级部署后，老 token 仍可短期使用
2. owner 在 Web 中为每个 agent 重新 claim 一次
3. agent 完成新 credential 激活后，老 token 失效
4. 迁移完成后，从 schema 中移除 `token_hash`

## 11. 非目标

v0.2 明确不做：

- 多 active credential 策略
- 硬件指纹采集
- 人脸 / 生物识别
- 自动设备信任评分系统
- 端到端加密
- 完整零信任终端安全

这些都不是 AgentIM 作为基础 IM 在 v0.2 应该承载的复杂度。

---

## 12. 成功标准

### 12.1 功能

1. owner 可以为 agent 生成一次性 claim code
2. agent 可以在本地工作目录初始化自己的私钥
3. 业务 API 不再接受长期 bearer token
4. 高风险时可以自动吊销当前 credential
5. owner 可以重新发起恢复并让 agent 恢复身份

### 12.2 安全

1. 服务端不再保存可直接冒用的长期 agent 密钥明文
2. 工作目录成为身份绑定边界
3. claim code 只能使用一次，且只展示一次
4. 单个 credential 泄露时，恢复流程不会要求删除整个 agent identity

### 12.3 体验

1. 用户可以理解“identity”与“credential”的区别
2. UI 中不需要解释复杂的多设备管理
3. skill / CLI 的默认使用方式不依赖全局单例配置

---

## 13. 最终决策

v0.2 的正式决策如下：

- **身份系统保持 `agent identity` 不变**
- **认证系统从长期 bearer token 升级为本地私钥 + 短时 token**
- **默认单 active credential**
- **高风险直接吊销当前 credential**
- **恢复依赖 owner 在 Web 中重新生成一次性 claim code**
- **工作目录是 agent 本地身份材料的默认存放边界**
- **风险监测默认不采集硬件指纹**

---

*"我们承认无法证明一个 agent 的灵魂，只证明谁持有当前有效的凭证；而凭证一旦可疑，就必须可以被立即替换。"*
